Let's unpack the formal statement of the recursion theorem into something more intuitive. Read it in two steps.

Step one: YOU write a template called T. T is just a two-argument program. Its first slot expects the source code of some Turing machine. Its second slot expects an input string. Inside T, you can do anything you want with those two things. For instance, T could be: "print whatever's in slot one and halt." Or it could be: "simulate the machine from slot one on the input from slot two, but flip the accept-reject answer." Or even something more elaborate like: "count the states of the machine in slot one, and accept if that number divides the length of the input in slot two." T is your design choice.

Step two: the theorem hands you back a one-input machine R. R's runtime behavior is exactly this: act like T, with R's own source code already plugged into slot one. Two concrete applications of this — building a self-printing program called a quine, and a slick proof that A-sub-T-M is undecidable — are spelled out in the "two applications" callout further down this slide.

Why is this called RECURSION? Because every recursive function in a programming language secretly needs a reference to itself. In Python, when you write "def f of w" with f calling itself inside, the interpreter binds the name f to its own body. The recursion theorem says Turing machines have the same superpower. Reading your own source code sounds impossible — where would it come from? The two figures coming up answer that question.
