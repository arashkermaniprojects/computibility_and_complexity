Let’s talk about the problem we’re facing with the language {0ⁿ1ⁿ}. This language requires that for every sequence of 0s, there’s an equal number of 1s following them. Now, on a single-tape Turing machine, you’re limited to just one head moving back and forth, which means you can’t count the 0s and 1s at the same time. You only have one option: to pair off each 0 with a corresponding 1.

The strategy here, as outlined in Sipser’s algorithm, involves marking a 0 and a 1 as you go. After marking a pair, you rewind back to the start and repeat the process. Each time you do this pairing, it takes O(n) time because you have to move all the way to the end of the input and then back. Since there are n/2 pairs to match, the overall time complexity ends up being Θ(n²).

There’s a lower bound to this as well. For the head to communicate between the 0s and 1s, it has to cross the middle of the input about Θ(n) times, confirming that the time taken can’t be better than quadratic. If we compare this to a two-tape Turing machine, it could solve the same problem in O(n) time. It just copies the 0s to a second tape and processes them in sync. This highlights the trade-off of using only one tape: you end up with that quadratic time cost.
