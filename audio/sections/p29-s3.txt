When working on the reduction from 3SAT to SUBSET-SUM, it’s crucial to pay attention to the base you choose. Using a base of four or higher is important because it allows each clause column to sum up to three, which is the maximum from three literals plus slack. If you were to use base 10, you wouldn’t have issues with overflow since the digit can handle those sums without carrying over. However, in binary, you'd encounter carries that could mess up other digit positions, which can lead to incorrect results.

Now, let’s talk about slack: each clause gets its own slack numbers, typically two bits of value one each. These slacks serve as padding, giving us flexibility to ensure that every clause sum is reachable. The numeric values we’re working with can be quite large, reaching up to 10 raised to the power of n plus m, where n is the number of variables and m is the number of clauses. While the number of digits is manageable and polynomial, the actual numeric value isn't, which introduces complexity.

When we apply dynamic programming to solve SUBSET-SUM, we often deal with a time complexity of O(n times t), but remember that t can grow exponentially with the input size. This means we're operating in pseudo-polynomial time, which doesn't contradict the P versus NP question. Understanding these pitfalls is essential to successfully navigating the reduction process.
