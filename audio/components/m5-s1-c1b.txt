This companion picture unpacks the three abstract terms the previous narration used — certificate, variable, and clause — by making each one a concrete object you can point at.

Panel one. Start with any NP language L. The defining property of NP is: there exists a polynomial-time Turing machine V, called the verifier, such that a string x belongs to L if and only if there is some short certificate c such that V running on the pair x, c accepts. The certificate c is the GUESS — it's the witness we're trying to find. In a SAT problem, c would be the truth assignment. In a clique problem, c would be the list of vertices forming the clique. In a Hamiltonian path problem, c would be the order in which to visit the vertices. The certificate is the answer we don't have yet.

Panel two. Now imagine V running on the pair x, c. As V works, its tape changes step by step. Stack the tape snapshots vertically — row zero is the initial tape, row one is the tape after one step, row two after two steps, and so on. You get a 2-D grid. That grid is the computation tableau — the big diagram above this one. Row zero shows the start configuration: the head in state q-zero at cell zero, then the input string x in the next few cells, then the certificate c filling the cells after that.

Panel three. Now the encoding move. For each cell of the tableau and for each possible tape symbol s, we make ONE Boolean variable, x-sub-t-p-s. This variable is TRUE if and only if cell t, p in the tableau holds the symbol s. So a single cell of the tableau corresponds to a small batch of Boolean variables — one for each possible symbol. The total variable count is roughly the number of cells times the alphabet size — polynomial in the input length.

Panel four. The constraints. Three clause families pin the Boolean variables down to exactly the assignments that correspond to valid runs of V. The START family forces row zero to be the correct start configuration. The MOVE family forces every 2-by-3 window in the tableau to be δ-consistent — one clause per window. The ACCEPT family is a single big OR-clause: somewhere in the tableau, some cell must hold the accept state q-accept. AND all three families together and you get phi-sub-x — the SAT formula for input x.

The bottom band reads the equivalence chain. Phi-x is satisfiable if and only if there is a truth assignment of the variables, if and only if there is a valid tableau for V on x, c, if and only if there exists a certificate c that makes V accept, if and only if x is in L. So finding the certificate IS finding a satisfying truth assignment. That's why solving SAT solves every NP language. That's why SAT is NP-hard. Combined with SAT being in NP, this is the Cook-Levin theorem.
