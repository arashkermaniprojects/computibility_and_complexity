Let’s dive into the diagonal recipe, which is a clever technique to show how more space gives us strictly more languages. We start by identifying all the Turing machines we want to outsmart, let’s call them M sub 1, M sub 2, and so on. These machines use at most some function f(n) space. We can list their descriptions in a lexicographical order, kind of like sorting words in a dictionary. 

Now, we build our special machine, D. Its job is to take an input that describes one of those machines, say ⟨M⟩, and simulate M on that same input. But here’s the twist: D is only allowed to use f(n) space, so it has to be careful. It uses a counter to keep track of its own space usage, and if M halts with some answer b, D will output 1 minus b. 

Now, let’s talk about D’s resources. It uses f(n) space for the simulated tape, plus an extra O(log f(n)) space for that counter. So overall, D uses O(f(n) log f(n)) space. This log factor comes from simulating M’s tape because D needs to track where M's head is, its current state, and how many steps it has taken. That’s why we have this extra overhead. Almost any reasonable function you can think of, like n, n squared, two to the n, or log n, is space-constructible, meaning there’s a Turing machine that can write out f(n) on its tape using O(f(n)) space.
