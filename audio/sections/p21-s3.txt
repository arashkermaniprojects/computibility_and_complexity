Let’s dive into a specific example of 3SAT, where we have four variables: x1, x2, x3, and x4. We're looking at a formula φ that consists of three clauses: the first clause is (x1 ∨ ¬x2 ∨ x3), the second is (¬x1 ∨ x2 ∨ x4), and the third is (x2 ∨ ¬x3 ∨ ¬x4). 

Now, if we want to find a satisfying assignment—basically, a way to set these variables to true or false to make φ true—we could try every possible combination of variable assignments. With four variables, there are 2 to the power of 4, which is 16 different combinations. For each of these combinations, we need to check all three clauses, leading us to a total of 48 clause evaluations. 

Now, imagine scaling this up to 30 variables. The number of assignments jumps to over a billion clause checks! 

On the other hand, if we have a potential solution, like τ equals (1, 0, 1, 0), our verifier can quickly check it. It evaluates the first clause and sees it’s true, but for the second clause, it finds that all parts are false, so it rejects τ after just nine evaluations. If we try τ equals (0, 1, 0, 0), it finds that this one satisfies all clauses and accepts it, again in nine evaluations. 

The key takeaway is that while the verifier can check a solution quickly, the brute-force solver's time explodes exponentially with the number of variables, which shows the asymmetry between finding a witness and verifying one.
