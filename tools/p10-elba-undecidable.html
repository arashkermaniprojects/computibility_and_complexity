ELBA is undecidable

Build an LBA BM,w that recognizes accepting computation histories of M on w.

CLO-2R1·30 ptsL7
1 · Theory

An LBA is a TM restricted to use only the cells of its input. Surprisingly: ALBA is decidable (configurations are finite-bounded) but ELBA is undecidable.

Reduction ATM ≤ ¬ELBA

Given ⟨M, w⟩, build LBA BM,w on input string x:

  1. Check x parses as C1 # C2 # … # Cℓ
  2. Check C1 is the start configuration of M on w
  3. Check each (Ci, Ci+1) follows by one δ-step of M
  4. Check Cℓ contains the accept state
  5. Accept x iff all checks pass.

L(BM,w) ≠ ∅ iff M accepts w. So a decider for ELBA decides ¬ATM. Impossible.

3 · Q&A defence notes
  • Q1. Why CAN an LBA check accepting computation histories?
    Each check is local: parsing, start-config match, adjacent δ-step validity, final-config accept-state check. All can be done by scanning the input multiple times within the original tape — that's the LBA budget.
  • Q2. Why is ALBA decidable but ELBA isn't?
    ALBA has fixed-length input; bounded configurations. ELBA asks "is the language empty?" over ALL inputs — that's a quantifier-over-all-strings question, much harder.
  • Q3. What's the size of an LBA computation history?
    If the LBA runs on input of length n, it has at most |Q|·n·|Γ|n distinct configurations. Repeats indicate looping. Hence the bound on ALBA.
