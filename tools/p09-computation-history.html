Computation history of a Turing machine

The trace C1 # C2 # … # Cℓ — the bridge from TMs to humbler models.

CLO-2R1·30 ptsL7
1 · Theory

A computation history of M on w is the full sequence of configurations the TM passes through, written as a single string with separators:

C1 # C2 # C3 # … # Cℓ
Local checkability

A string is "an accepting computation history of M on w" iff it satisfies local properties: (a) the first config is the start; (b) each adjacent pair (Ci, Ci+1) follows by one transition of δ; (c) the last config has the accept state. Humble machines (LBAs, CFGs) can verify these.

This is the key technique behind ELBA and ALLCFG undecidability (Problems 10, 11).

3 · Q&A defence notes
  • Q1. Why insert the state inside the tape?
    A configuration must capture tape contents, head position, and state in one string. Writing the state right before the head cell encodes all three compactly.
  • Q2. Why is local checkability so powerful?
    LBAs / CFGs can verify a 2×3 "window" between adjacent configs but cannot count globally. Local checking is in their power; full simulation isn't.
  • Q3. Why doesn't this trick work for proving ATM undecidable?
    It does the opposite: it shows certain TM questions reduce to questions about LBAs/CFGs. The undecidability flows the other way.
  • Q4. What if the TM loops?
    No finite accepting computation history exists. The reduction's target language is empty in that case, which is exactly the information we want to encode.
