This callout answers four common questions students ask when they first see a computation history table.

First: why does a single row mix the state and the tape contents? Each row is one CONFIGURATION — Sipser's compact snapshot of the whole machine at one moment. We write the tape contents in order with the state symbol inserted at the head's position. So zero q-one one one zero means the tape reads zero one one zero left to right, the head is over the leftmost one, and the state is q-one. The state isn't literally on the tape — it's notation that lets one string encode the entire configuration.

Second: why does the state column move forward over time? Because the state symbol sits where the head sits, and each transition moves the head left or right by one cell. Watching the column shift IS watching the head move.

Third: why exactly a 2-by-3 window — not 3-by-4 or 3-by-2? A 2-by-3 window is the smallest rectangle that fully determines whether one transition is legal. Two rows because you need a before-row and an after-row to check a transition. Three columns because the head can move at most one cell, so the next row's middle cell depends only on three neighboring cells of the previous row. 3-by-4 would work but waste space. 3-by-2 doesn't work — two columns can't distinguish whether the head came from the left or the right.

Fourth: why does that make 2-by-3 a validity indicator? Local-to-global. If every 2-by-3 window obeys delta, the entire tableau is a valid computation; one bad window kills validity. This is the lever that Cook-Levin, E sub L B A, and ALL sub C F G all pull.
